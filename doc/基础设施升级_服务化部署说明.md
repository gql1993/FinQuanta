# FinQuanta 基础设施升级说明

## 本轮目标

本轮选择的是路线 `B`，即把当前以本地 SQLite + 桌面端为主的架构，逐步升级为可服务化、可部署、可扩展的后端平台基础设施。

本次已完成的内容聚焦在五件事：

1. 为 PostgreSQL 迁移准备初始化脚本与迁移入口。
2. 为 Redis 快照缓存准备适配骨架。
3. 为 API 服务化部署准备 Docker / Nginx 运行雏形。
4. 将 API 关键查询改为走统一仓储层 `repo`。
5. 增加第一版用户管理接口与标准启动入口。

## 数据与部署策略（推荐：模式 A）

与「基础设施路线 B」并行，产品侧推荐 **模式 A**：

- **API / 服务化部署**：使用 PostgreSQL（`FINQUANTA_DB_BACKEND=postgres` + `FINQUANTA_POSTGRES_DSN`），便于多端访问、备份与运维。
- **桌面客户端**：默认继续使用本地 SQLite（`data_cache/quant.db`），安装即用、不强依赖网络；**勿在桌面环境误配为 PostgreSQL**，除非明确需要桌面直连同一实例做联调。
- **数据关系**：此模式下 API 与桌面**不要求**天然一致；若将来需要一致，应通过单独的同步、导入导出或产品规则设计，而不是默认假设两端共库。

说明已写入仓库根目录 `.env.api.example` 文件头注释。

**与「本轮目标 / 路线 B」的关系**：路线 **B** 指基础设施升级方向（服务化、PostgreSQL 迁移脚手架、可部署）；**模式 A** 指数据产品策略（桌面默认 SQLite、API 用 PostgreSQL、两端不要求共库）。二者并行，并不矛盾。

## 已新增文件

### PostgreSQL 迁移脚手架

- `infra/postgres_init.sql`
- `infra/migrate_sqlite_to_postgres.py`

作用：

- 先把关键核心表定义成 PostgreSQL 可落地的结构。
- 提供一个 SQLite 数据导出入口，后续可以扩展为真正的增量/全量迁移程序。
- 当前仍不强制切换数据库，保持桌面端与现有流程稳定。
- `PostgresRepository` 已从占位版升级为可实际连接的实现，支持 `fetchone/fetchall/execute/executescript/kv_get/kv_set`。

### Redis 缓存适配骨架

- `api_server/cache_provider.py`

作用：

- 对系统快照增加 Redis 可选缓存层。
- 若未配置 `FINQUANTA_REDIS_URL`，系统仍保持原有逻辑，不会报错。
- 若配置 Redis，则 `/api/snapshot/system` 可以优先走缓存，减轻高频读取压力。

### 服务化部署雏形

- `api_server/Dockerfile`
- `deploy/docker-compose.yml`
- `deploy/nginx.conf`
- `start_api.bat`
- `.env.api.example`
- `check_api.bat`
- `infra/check_backend_ready.py`
- `smoke_api.bat`
- `infra/smoke_api.py`
- `validate_postgres.bat`
- `infra/validate_postgres_backend.py`
- `migrate_import.bat`

作用：

- 允许以容器方式启动 `FastAPI + PostgreSQL + Redis + Nginx`。
- 为后续 Web 版、小程序版、远程多端访问提供统一 API 出口。
- 提供一键环境自检入口，便于部署前快速确认数据库与缓存状态。
- 提供关键接口冒烟测试入口，便于联调和回归验证。
- 提供 PostgreSQL 后端切换验证入口。

## 已做代码增强

### 配置中心增强

`api_server/config.py` 新增：

- `FINQUANTA_ENV`
- `FINQUANTA_API_BASE`
- `FINQUANTA_SNAPSHOT_CACHE_TTL`

这意味着后端环境配置开始从“写死”逐步转向“环境变量驱动”。

### `.env` 自动加载

新增：

- `api_server/env_loader.py`

启动 API 时会优先尝试读取：

1. `.env.api.local`
2. `.env.api`
3. `.env`

并使用 `os.environ.setdefault()` 注入环境变量。

这意味着：

- 终端里显式传入的环境变量优先级最高
- 本地/测试/生产可以通过不同 `.env` 文件做轻量切换
- Web 端默认 API 地址也可通过 `FINQUANTA_API_BASE` 统一配置

### API 健康检查增强

`/health` 现在会返回：

- 当前运行环境
- 当前数据库后端
- Redis 缓存是否启用

这有助于部署后快速排查问题。

### 系统快照缓存

`/api/snapshot/system` 现在会：

1. 先查 Redis 缓存
2. 未命中时读取本地快照
3. 再回写缓存

这样后续小程序/Web 高频访问首页时，压力会更小。

### 中台层数据访问收口

新增：

- `desktop/platform_store.py`

当前已先接入两类中台模块：

- `desktop/task_orchestrator.py`
- `desktop/snapshot_service.py`

意义：

- 任务日志、系统事件、系统快照不再各自直连底层存储细节
- 后续如果要继续抽象成统一 service/repository，会更平滑
- API、Web、OpenClaw 对中台数据的复用成本更低

### 仓储层统一

`api_server/main.py` 中以下接口已经改为通过统一仓储层读取，而不是直接写死 `sqlite3`：

- `/api/scan/latest`
- `/api/portfolio/recommendations`
- `/api/stock/{code}`
- `/api/stock/{code}/kline`

这样当后端从 SQLite 切到 PostgreSQL 时，这些 API 不需要再重写一遍。

### 用户管理接口

新增了第一版后台管理接口：

- `GET /api/admin/users`
- `POST /api/admin/users`
- `DELETE /api/admin/users/{username}`
- `POST /api/admin/tokens/revoke`
- `GET /api/admin/auth-audit`

当前可完成：

- 查看用户列表
- 创建/更新用户
- 删除非 `admin` 用户
- 强制某个用户的所有 Token 失效
- 查看最近认证审计日志

权限规则：

- 仅拥有 `settings:write` 权限的用户可调用
- 当前默认 `admin` 用户保留

### 认证安全增强

当前认证层已经补充：

- 密码哈希存储：使用 `PBKDF2-SHA256`
- 老账号兼容迁移：若历史密码是明文，用户成功登录后会自动升级为哈希
- 主动退出：`POST /api/auth/logout`
- 修改密码：`POST /api/auth/change-password`
- 修改密码后自动使旧 Token 失效
- 认证审计：登录、登出、改密、删用户、撤销 Token 均会记录

### 依赖健康检查

新增：

- `GET /health/deps`

可用于检查：

- 数据库是否连通
- Redis 缓存是否连通或未启用

### SQLite 全量迁移到 PostgreSQL

1. 导出本地 `data_cache/quant.db` 为 JSON：

```bash
python infra/migrate_sqlite_to_postgres.py export
```

2. 在目标 PostgreSQL 上执行建表（脚本内会自动执行 `infra/postgres_init.sql`，并补全旧库缺失列），再导入快照：

```bash
set FINQUANTA_POSTGRES_DSN=postgresql://user:pass@127.0.0.1:5432/dbname
python infra/migrate_sqlite_to_postgres.py import --snapshot infra/sqlite_export_snapshot.json
```

或 `migrate_import.bat`（依赖同上环境变量）。默认会先 `TRUNCATE` 目标库中相关表再写入；若需追加写入而非清空，可加 `--no-truncate`。

- `daily_kline` 行数较多时，导入会自动使用 PostgreSQL `COPY`（默认行数阈值 5000，可用 `--copy-threshold N` 调整）。
- 若暂时不迁移 K 线等大表，可 `--skip-table daily_kline`：该表既不会清空也不会写入，便于先迁配置与业务小表。

3. 切换 API 使用 PostgreSQL：设置 `FINQUANTA_DB_BACKEND=postgres` 与 `FINQUANTA_POSTGRES_DSN`，必要时生产环境勿使用 `FINQUANTA_PG_PERSISTENT`（该模式主要为 PGlite/验证场景设计）。

## 当前阶段的定位

这次不是“一步切换到 PostgreSQL/Redis 正式生产版”，而是先搭一个不会破坏现有系统的过渡层。

也就是说，当前状态是：

- 桌面端仍可继续以本地 SQLite 运行
- API 端已经具备后续迁移的接口结构
- Web/小程序后续可以逐步切到真正的服务化模式

## 下一步建议

建议按下面顺序推进：

1. 把 `snapshot_service`、`task_orchestrator`、`messages` 等核心读取逻辑继续逐步改为走 repository。
2. 增加角色变更审计、Token 有效期配置、密码复杂度校验。
3. 增加 `.env` 自动加载、部署检查脚本、数据库初始化命令。
4. 让 Web 端优先 API 模式彻底落地，小程序进入真机联调。
5. 逐步推进 PostgreSQL 正式切换与数据迁移验证。

## 备注

当前新增内容均为“兼容式增强”，原则是不影响现有桌面客户端主链路。

如果继续走路线 `B`，我建议下一步就直接做：

`权限中心深化 + 仓储层进一步收口 + API 运行环境标准化`
