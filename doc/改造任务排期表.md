# FinQuanta 改造任务排期表

本文档用于把前述改造方案落成可执行排期，目标是确保 **系统持续可运行、边改边收口、避免大爆炸式重构**。

适用范围：

- 当前基线版本：`f771a1e`
- 参考基线文档：`doc/改造前备份与基线清单.md`
- 不纳入改造范围：`claude-code-main/`

---

## 一、排期原则

### 1. 先收口，再扩展

改造顺序固定为：

1. 文档与边界冻结
2. 应用服务层收口
3. AI 执行框架与风控审批
4. 数据模式收口
5. 插件化、灰度、可观测性补齐

### 2. 每阶段都必须可运行

每个阶段结束时，至少满足以下之一：

- 桌面主链路可运行
- API 主链路可运行
- Web 主链路可运行

禁止“先大拆，最后再修”。

### 3. 新需求默认暂停接入核心层

在 `M1-M3` 期间：

- 原则上不往 `desktop/main_window.py`
- 不往 `api_server/main.py`
- 不往 `web_app.py`

继续堆新的核心业务逻辑。

### 4. 所有改造优先沉淀为中间层

优先新增：

- `core/application/*`
- `core/ai/*`
- `core/risk/*`
- `core/repositories/*`

而不是继续加大既有入口文件复杂度。

为便于排期，建议按角色分工：

| 角色 | 主要负责 |
|------|----------|
| 架构/平台负责人 | 边界冻结、目录规划、服务层抽象、注册中心、配置中心 |
| 后端负责人 | `api_server` 改造、`core/application`、`repositories`、运行模式 |
| 桌面负责人 | `desktop/panels/*`、`desktop/main_window.py`、桌面适配层 |

| 风控负责人 | 审批流、执行权限、规则抽象、风险策略 |
| 测试/联调负责人 | 冒烟验证、回归清单、运行基线记录 |

## 六、关键风险与控制

| 风险 | 表现 | 控制策略 |
|------|------|----------|
| 范围漂移 | 还没收口就继续加功能 | 所有新增核心功能先冻结或挂起 |
| 入口文件继续膨胀 | `main_window.py`、`api_server/main.py` 继续变大 | 新逻辑只允许进 `core/*` |
| 改造导致不可运行 | 重构完成前主链路中断 | 每阶段设置冒烟门槛 |
| 双模式长期共存 | 本地模式和平台模式边界继续模糊 | 在 `M3` 明确正式策略 |
| AI 能力继续分叉 | 助手、AI 仓、OpenClaw 各搞一套 | `M2` 统一上下文和决策入口 |

## 九、结论

本排期表的核心目标不是“把代码拆漂亮”，而是让 FinQuanta 从：

- 桌面主程序承载核心业务

逐步演进为：

- 多端共用应用服务层
