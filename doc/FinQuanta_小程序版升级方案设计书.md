# FinQuanta 小程序版升级方案设计书

## 一、目标定义

### 1.1 建设目标

在现有 FinQuanta 桌面版与 Web 版基础上，规划一套可落地的**微信小程序版升级路径**，实现：

1. 朋友或团队成员可通过微信远程访问系统核心能力。
2. 保持桌面版作为“重操作端”，小程序作为“轻操作端”。
3. 保证桌面版、小程序版、Web 版共享同一套核心数据和调度体系。
4. 将现有单机架构逐步演进为“多端共享 + 后端统一服务 + 可持续运营”的平台架构。

### 1.2 总体定位

小程序版不是桌面版的简单缩小，而是**远程运营入口**：

- 桌面版：策略研究、参数配置、复杂操作
- Web 版：远程查看、运营监控、任务触发
- 小程序版：移动查看、审批确认、预警接收、轻量操作

### 1.3 设计原则

1. 先服务化，再小程序化
2. 先高频功能，再复杂功能
3. 先查看和审批，再做深度操作
4. 三端共用同一数据层和任务层

---

## 二、现状评估

### 2.1 当前系统现状

当前 FinQuanta 已具备以下能力：

1. 桌面端 PyQt6 主应用
2. Web 版 Streamlit 远程访问能力
3. OpenClaw 智能中枢
4. Daemon 自动化调度
5. SQLite 本地数据库
6. 统一快照 `system_snapshot`
7. 统一市场状态机 `market_state`
8. 统一任务审计与系统事件日志

### 2.2 为什么不能直接改成小程序

当前架构中，以下部分无法直接搬到微信小程序：

1. PyQt6 界面无法运行在微信环境中
2. 本地 SQLite 不能被微信直接访问
3. 本地 daemon/OpenClaw 不能被微信直接调度
4. 小程序必须通过 HTTPS 接口访问服务端
5. 微信小程序需要合法域名、服务器和账号审核

因此，小程序版本质上需要：

> **前端小程序 + 后端 API 服务 + 统一数据库 + 云端任务调度**

## 三、升级总体路线

### 3.1 建议路线

```text
当前：桌面版 + 本地SQLite + 本地Daemon
        ↓
阶段A：抽离后端服务层（API）
        ↓
阶段B：数据库服务化（PostgreSQL/Redis）
        ↓
阶段C：开发微信小程序前端
        ↓
阶段D：接入登录/权限/推送/审批
        ↓
阶段E：形成桌面端 + Web端 + 小程序 三端协同平台
```

### 3.2 升级策略

采用**平滑迁移**而不是推倒重建：

1. 桌面版继续保留作为主控端
2. 先把核心能力抽成 API
3. Web 版从“展示端”升级为“业务端”
4. 小程序最后接入

这样风险最小，能持续交付，不影响现有系统使用。

### 4.1 升级后的总体架构

```text
┌────────────────────────────────────────────────────────────┐
│                       客户端层                              │
│  PyQt6桌面端   |   Web端(H5/Browser)   |   微信小程序端     │
└────────────────────────────────────────────────────────────┘
                          ↓
┌────────────────────────────────────────────────────────────┐
│                      API 服务层                             │
│ auth_api | snapshot_api | portfolio_api | scan_api         │
│ openclaw_api | analysis_api | backtest_api | settings_api  │
└────────────────────────────────────────────────────────────┘
                          ↓
┌────────────────────────────────────────────────────────────┐
│                     核心业务层                              │
│  选股引擎 | 量子优化 | AI决策 | OpenClaw | 风控引擎 | 学习引擎 │
└────────────────────────────────────────────────────────────┘
                          ↓
┌────────────────────────────────────────────────────────────┐
│                     调度与消息层                            │
│  Scheduler | Task Orchestrator | Push Gateway | Alert Hub  │
└────────────────────────────────────────────────────────────┘
                          ↓
┌────────────────────────────────────────────────────────────┐
│                     数据与缓存层                            │
│ PostgreSQL（主库） | Redis（快照/缓存） | 对象存储（文档）   │

```

### 4.2 三端职责划分

| 端 | 定位 | 主要功能 |
|---|---|---|
| 桌面端 | 专业操作端 | 复杂配置、策略研究、量子优化、图表分析 |
| Web端 | 管理运营端 | 运行中心、日志中心、远程触发任务 |
| 小程序 | 轻量访问端 | 查看总览、审批交易、接收预警、查看个股、查看策略结果 |

## 七、数据库迁移设计

### 7.1 为什么需要迁移

当前 SQLite 适合单机，但小程序版要求：

1. 多设备同时访问
2. 后端服务多进程运行
3. 长期稳定保存
4. 任务调度与 Web 并发访问

所以建议升级为：

- **PostgreSQL**：主库
- **Redis**：缓存与快照
- **对象存储**：文档/PPT/报表

### 7.2 迁移方案

#### 主库（PostgreSQL）

迁移以下核心表：

- `daily_kline`
- `stock_list`
- `board_stocks`
- `ai_positions`
- `ai_trade_log`
- `trend_verify`
- `ai_decision_memory`
- `system_event_log`
- `task_run_log`
- `operation_log`
- `daily_nav`

#### Redis 缓存

缓存以下高频数据：

- `system_snapshot`
- `market_state_snapshot`
- `portfolio_risk`
- `last_scan_results`
- `strategy_rotation`
- `sector_rotation`

### 7.3 迁移路径

1. 保留 SQLite，不中断桌面版
2. 增加 PostgreSQL 适配层
3. 后端 API 先读 SQLite，逐步切换为读 PostgreSQL
4. 最终三端共用 PostgreSQL + Redis

## 八、调度与消息迁移设计

### 8.1 调度层升级

当前 `daemon_scheduler.py` 可平滑演进为服务端调度器。

建议：

- 本地版：继续保留 daemon
- 云端版：迁移到 `APScheduler` / `Celery Beat`

### 8.2 消息推送层

小程序版建议保留三层推送：

1. **企业微信**：运营预警、任务通知
2. **小程序订阅消息**：用户级提醒
3. **站内消息中心**：统一查看

## 九、开发排期建议

### 阶段A：后端服务化（1-2周）

目标：

1. 建立 FastAPI 服务
2. 封装 snapshot / portfolio / scan / openclaw / settings API
3. 保持桌面版继续运行

### 阶段B：数据库迁移（1周）

目标：

1. SQLite → PostgreSQL 迁移脚本
2. Redis 接快照缓存
3. 校验三端一致性

### 阶段C：小程序 MVP（2-3周）

目标：

1. 首页总览
2. 选股结果
3. AI 仓查看
4. OpenClaw 运行中心
5. 个股分析
6. 消息中心

### 阶段D：审批与触发（1-2周）

目标：

1. 小程序确认买卖
2. 远程触发扫描/学习/全流程
3. 多用户权限

---

## 十、建议的落地顺序

### 最优顺序

1. **先做 API 层**
2. **再做数据库迁移**
3. **再做微信小程序 MVP**
4. **最后做权限与多人协作**

### 为什么这样做

因为当前桌面版与 Web 版已经具备强大的业务逻辑，只要把逻辑从“本地调用”抽成“服务调用”，小程序版就能复用 70%-80% 的核心能力，而无需重写业务本身。

---

## 十一、结论

从当前 FinQuanta 的成熟度看，系统已经具备升级为微信小程序版的基础条件：

1. 已有统一快照服务
2. 已有统一任务编排与审计
3. 已有 OpenClaw 中枢
4. 已有 Web 版雏形
5. 已有完整业务闭环

因此，**当前最合理的下一步不是直接开发小程序页面，而是先把现有系统抽成 API 平台**。  
完成这一步后，小程序版就可以作为轻量入口快速落地。
