## 二、技术领域

本发明涉及金融科技与计算机软件领域，尤其涉及面向证券（以 A 股为主要场景）的量化交易客户端、后台任务调度、大语言模型（LLM）辅助决策、组合与量子启发式优化、以及可服务化部署的 API 与多端数据访问架构。

---

## 三、背景技术

传统量化系统常见局限包括：

1. **策略与执行割裂**：选股、回测、模拟交易分散在不同工具，难以在同一账户视图下对比多策略、多仓位表现。  
2. **LLM 直接用于报价的风险**：若将模型输出价格作为成交依据，易产生与真实行情不符的“价格幻觉”，导致模拟失真或误导。  
3. **自动化缺乏可审计闭环**：定时任务与手工操作日志分散，故障与合规追溯困难。  
4. **单机与多端矛盾**：桌面端强调离线可用与本地库；Web/小程序强调集中服务与高并发读，若强行共库易导致部署与一致性成本过高。  
5. **智能体“可执行”与风控**：自然语言指令若不经权限校验与确认，存在误操作与安全风险。

因此，需要一种在**统一产品内**整合多策略引擎、AI 决策、组合优化、风控与执行抽象，并具备**统一快照、任务审计、可选服务化出口**的技术方案。

## 四、发明内容

### 4.1 要解决的技术问题

本发明旨在至少解决以下问题：

- 在单一桌面客户端内完成**数据采集 → 多策略选股 → 回测与验证 → 多模式模拟交易 → 消息推送 → 绩效与归因**的闭环；  
- 对大语言模型参与的交易决策，**强制以真实行情或本地 K 线校验后的价格**作为下单与持仓估值依据，抑制价格幻觉；  
- 通过**统一快照服务**与**任务编排/系统事件**机制，为 UI、守护进程、OpenClaw 管线及远程 API 提供一致只读视图与审计轨迹；  
- 通过**交易主引擎**聚合订单管理、风控、账户与网关（模拟/实盘可切换），使执行路径清晰可扩展；  
- 通过**服务化 API** 与 **SQLite / PostgreSQL 双轨仓储**、**可选 Redis 快照缓存**，在不要求桌面与云端天然数据一致的前提下，支持多端接入与运维扩展。

### 4.2 技术方案概述

本发明提供一种证券量化交易系统，在逻辑上包括以下组成部分（可与实际模块名对应）：

#### （1）表现层：PyQt6 多标签工作台

主窗口采用侧边标签组织功能，典型包括：总览、选股（内嵌走势验证与回测子页）、短期选股、手动仓、AI 模拟仓、AI 助手、OpenClaw、运行中心、个股分析、设置等。图表类视图可通过嵌入式 Web 引擎加载本地 HTML（如 Plotly.js）实现交互式 K 线。

#### （2）业务策略层

- **多策略选股**：集成多种经典与本土化策略（如趋势/VCP、CANSLIM、海龟、价值/动量、国内趋势与价值等），并行产出候选标的。  
- **量子启发式与组合优化**：在策略融合或仓位分配场景下，提供量子近似优化算法（如 QAOA 经典模拟）、模拟退火、禁忌搜索及经典均值方差、风险平价等，用于权重或选股集优化。  
- **事件与基金维度短期选股**：基于事件描述匹配板块与标的，或基于基金持仓披露做跟踪分析。  
- **走势验证**：记录选股信号并跟踪多周期实际涨跌，用于校准预测与策略质量评估。

#### （3）AI 模拟仓的多模式并行

设置多种并行模拟仓位模式，例如：完全自主、半自主（建议后人工触发）、推荐仓、自定义规则仓、量子优化仓等。各仓独立状态与流水，支持对比展示。

**关键约束**：买入/卖出等涉及成交价的操作，优先使用实时行情接口报价，其次回退至日 K 等本地可验证数据，并与模型输出解耦，从而形成“模型出意图、系统定价格”的机制。

#### （4）OpenClaw 智能管线引擎

将感知采集、因子研究、多智能体研判、组合优化、风控检查、执行、推送、归因、自主学习等环节串联为可编排管线。管线执行与系统其他任务统一记入任务运行日志；自主学习模块可根据历史表现调整策略权重等参数，形成反馈闭环。

#### （5）统一快照与运行中心

**快照服务**聚合：手动仓与多 AI 仓权益与现金、风险指标（如 VaR）、市场状态（如基于指数 regime 的判断）、最近任务运行记录、最近系统事件、以及交易主引擎状态摘要等，写入键值存储并供各模块读取。

**运行中心**在桌面集中展示上述快照摘要、近期任务与系统事件表，便于运维与排障。远程 **API** 暴露同源或子集数据（如 `/api/snapshot/system`、`/api/ops/tasks`、`/api/ops/events`），并可通过 Redis 对高频只读快照做短期缓存。

#### （6）任务编排与平台存储

通过任务编排模块包装定时任务与流水线：统一记录任务名、触发来源、耗时、状态与摘要；系统事件记录来源、分类、级别与详情。底层经平台存储抽象，避免各业务直接散落访问连接细节，便于 SQLite 与 PostgreSQL 双后端切换时保持一致写入路径。

#### （7）系统助手：意图—权限—审计—确认

对用户自然语言输入进行意图解析；按策略校验权限与风险等级；生成可预览的执行说明；对高风险或修改类操作要求用户确认后执行；全过程写入审计日志。未匹配意图时可回退到通用对话。该机制可与 API 侧助手服务共用同一套业务逻辑。

#### （8）交易主引擎（MainEngine）

参考专业交易框架思路，聚合：

- **事件引擎**：用于订单与系统事件的分发扩展；  
- **订单管理服务（OMS）**：统一提交、撤单与状态跟踪；  
- **风控模块**：在提交路径上实施规则拦截；  
- **账户服务**：查询资金与持仓；  
- **网关**：纸交易与实盘网关可配置切换（如环境变量控制），便于同一套上层逻辑在模拟与真实环境复用。

快照服务可读取主引擎摘要，使 UI 与 API 对“当前交易通道状态”有一致认知。

#### （9）后台守护调度器

以独立线程或独立进程方式运行调度表，按交易日时刻触发：行情拉取、K 线增量更新、板块与选股扫描、短期策略、多仓 AI 决策、止损与预警、走势验证、日报推送、自主学习、周期性回测与数据清理等。支持任务键级禁用与时间覆盖配置，并将执行纳入统一任务日志。

#### （10）数据层与部署模式

- **桌面默认**：本地 SQLite，WAL 模式与繁忙等待参数降低 UI 与后台并发锁冲突。  
- **API 部署**：可选 PostgreSQL 作为关系库；统一仓储层封装查询与写入；关键 API 不直接硬编码连接。  
- **缓存**：可选 Redis 仅用于快照等只读热点，未配置时自动降级。  
- **策略说明**：桌面与云端**不强制共库**；若需一致，通过同步或导入导出等产品化手段实现，降低默认部署复杂度。

#### （11）对外服务化 API

基于 FastAPI 提供健康检查、鉴权（用户/令牌/角色占位或可扩展）、系统快照、运行中心、组合摘要、OpenClaw 触发与自主学习触发等接口，便于 Web、小程序与远程运维接入。CORS 与多环境配置通过环境变量驱动。

### 4.3 有益效果

1. **端到端一体化**：同一产品覆盖策略研发、验证、模拟交易与消息触达，降低工具链切换成本。  
2. **AI 可用性增强**：通过行情强制绑定，显著提高模拟交易与复盘的可信度。  
3. **可观测与可审计**：快照 + 任务 + 事件 + 助手动作日志，形成运维与合规友好轨迹。  
4. **架构可演进**：主引擎与仓储抽象便于接入真实券商、扩展风控规则与多账户。  
5. **部署弹性**：本地离线可用与服务化多端接入并存，通过“不默认共库”控制一致性成本。

## 六、具体实施方式

以下结合实施例说明，本领域技术人员可对模块名、时间参数与策略集合作等同替换，而不脱离本发明精神。

### 6.1 实施例一：桌面客户端启动与数据初始化

用户启动客户端后，初始化本地数据库；可选将历史 CSV/JSON 缓存导入 SQLite；主窗口构建多标签界面；延迟启动守护调度线程并读取用户禁用的任务键与时间覆盖；启动周期 UI 刷新与后台 K 线批量更新（如对持仓与板块成分股限数量、多线程拉取）。

### 6.2 实施例二：多策略选股与回测

用户在选股面板选择策略与参数，系统从本地 `daily_kline` 等表读取 OHLCV，执行策略筛选；回测与走势验证可作为选股面板的子标签，调用本地回测引擎支持蒙特卡洛、滚动窗口与多策略对比，结果以表格与权益曲线展示。

### 6.3 实施例三：AI 模拟仓下单的价格约束

当 LLM 返回买卖意图与标的时，系统根据标的代码请求实时行情；若无实时数据则使用最近日 K 收盘价等；若与模型给出价格偏差超过阈值则修正或记录告警。持仓与流水写入 AI 仓相关表，供总览与快照聚合。

### 6.4 实施例四：统一快照构建

定时或事件触发调用快照构建函数：汇总手动仓 JSON、各 AI 仓状态、风险 KV、市场状态模块输出、最近任务与事件列表、主引擎 `snapshot()`；序列化写入 `system_snapshot` 供仪表盘、运行中心与 API 读取。

### 6.5 实施例五：OpenClaw 一键管线

用户或远程 POST 请求触发管线：按引擎实现依次调用数据采集、因子与智能体、组合优化、风控、执行、推送与学习评估；每步可记录系统事件；失败时中断并写日志。干跑（dry_run）模式用于演练不产生真实副作用（依实现定义）。

### 6.6 实施例六：系统助手确认执行

用户输入“查询持仓”类意图可直接返回快照子集；“触发任务”或“修改配置”类意图生成预览卡片，用户确认后执行 `dispatch_intent`，并更新动作状态与审计日志。

### 6.7 实施例七：API 与 Redis 缓存

部署时设置 PostgreSQL DSN 与 Redis URL。请求 `/api/snapshot/system` 时先查 Redis TTL 内缓存；未命中则构建或读取快照并回写。健康检查接口返回数据库后端与缓存开关状态便于监控。

### 6.8 实施例八：双轨数据访问

业务模块通过 `data_access` 或 `repo` 写入日 K、板块关系等；在 PostgreSQL 下使用 `ON CONFLICT` 语义，在 SQLite 下使用 `INSERT OR REPLACE` 等，对上层透明，减少迁移成本。
